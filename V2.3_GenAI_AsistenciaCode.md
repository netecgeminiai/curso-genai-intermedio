### [00:08-01:00] INTRODUCCIÓN A LA IA EN EL IDE
**[VISUAL: Pantalla mostrando a un programador escribiendo un comentario de código y la IA autocompletando un bloque entero (GitHub Copilot)]**

Hola. Hasta ahora hemos hablado de cómo poner Inteligencia Artificial dentro de las aplicaciones que le vendemos a los clientes. 

Pero, ¿qué pasa con nosotros? ¿Cómo usamos la IA en nuestro propio ciclo de desarrollo de software?

Hoy hablaremos de la Generación y Asistencia de Código. Las herramientas basadas en LLMs (como GitHub Copilot, Cursor, o los modelos de OpenAI y Claude) no están aquí para quitarte el trabajo. Están aquí para quitarte el trabajo *aburrido*, como escribir *boilerplate*, crear tests unitarios o cazar un bug minúsculo de sintaxis a las 3 de la mañana.

---

### [01:00-02:00] AUTOCOMPLETADO Y REFACTORIZACIÓN
**[VISUAL: Código "Spaghetti" feo transformándose en código limpio con un atajo de teclado]**

El primer nivel de uso es la **Programación en Pareja (Pair Programming)**. 

En lugar de ir a StackOverflow a buscar cómo se invierte una lista en Python o cómo hacer una conexión SQL, herramientas como Copilot leen el contexto de todos los archivos que tienes abiertos en tu editor. Escribes un comentario, y el modelo escribe la función por ti.

Pero el valor real para los desarrolladores intermedios está en la **Refactorización**.
Puedes seleccionar un bloque de código viejo, complejo e ineficiente de tu empresa, enviárselo a Claude o a GPT-4 y pedirle: *"Optimiza este bloque para reducir la complejidad ciclomática y agrega tipado estático (Type Hints)"*.

En segundos, pasas de código heredado (Legacy) a código moderno y mantenible.

---

### [02:00-03:10] EL RIESGO DE SEGURIDAD (ALUCINACIONES DE CÓDIGO)
**[VISUAL: Icono de alerta roja. Código generado por IA apuntando a una librería falsa o exponiendo una API Key]**

Pero cuidado, porque la magia tiene un precio oscuro. La IA alucina código.

Si le pides que use una función muy específica, el modelo podría *inventarse* el nombre de un paquete o una librería que no existe (esto se llama Hallucinated Packages). Si un atacante descubre qué paquetes inventa el modelo y los sube a NPM o a PyPI con código malicioso, tú lo instalarás sin darte cuenta y comprometerás tu sistema.

Además, los modelos a menudo proponen soluciones de código que funcionan, pero que introducen **Vulnerabilidades de Seguridad** (como inyecciones SQL o falta de validación de entradas). 

Por eso la regla de oro del desarrollador GenAI es: **La IA escribe, el Humano Audita y Verifica.** Nunca hagas un *commit* de código que no entiendas línea por línea.

---

### [03:10-04:00] CODE REVIEW AUTOMATIZADO
**[VISUAL: Diagrama de un Pull Request en GitHub -> Webhook a IA -> Aprobación/Rechazo automático]**

El uso más maduro de estas APIs es implementarlas en el pipeline de Integración Continua (CI/CD) para hacer **Code Reviews automatizados**.

Imagina esto: Un desarrollador junior hace un *Pull Request* (PR). Antes de que un desarrollador senior pierda tiempo revisándolo, un script en Python toma los cambios (el diff), se los envía a la API de OpenAI y le pide: *"Actúa como un arquitecto de seguridad. Revisa este código y dime si hay vulnerabilidades o si viola nuestros estándares de Clean Code"*.

El LLM comenta automáticamente el PR señalando errores.
