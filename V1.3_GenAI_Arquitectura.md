### [00:08-01:10] INTRODUCCIÓN AL PARADIGMA
**[VISUAL: Comparación de arquitecturas. Izquierda: MVC tradicional (App -> Servidor -> DB). Derecha: App -> Backend -> LLM Orchestrator -> VectorDB / Tools]**

Hola. En el desarrollo tradicional de software usábamos el patrón Modelo-Vista-Controlador (MVC). Teníamos una base de datos relacional (SQL), lógica de negocio en el backend y una interfaz de usuario. Era predecible.

La Inteligencia Artificial Generativa rompe este paradigma por completo.
Los Grandes Modelos de Lenguaje (LLMs) son poderosos, pero tienen tres grandes problemas de diseño:
1. Tienen "alucinaciones" (se inventan datos).
2. Son "Staateless" (no tienen memoria, olvidan el chat apenas termina la petición).
3. No saben qué pasó en el mundo después de la fecha de su entrenamiento.

Para solucionar esto, no modificamos el LLM. Construimos una **Arquitectura Envolvente** a su alrededor.

### [01:10-02:20] LOS 4 COMPONENTES DE LA NUEVA ARQUITECTURA
**[VISUAL: Diagrama construyéndose paso a paso mientras se mencionan los componentes]**

Una arquitectura moderna GenAI de nivel empresarial tiene cuatro capas críticas:

1. **La Capa de Orquestación (El Cerebro):**
No llamas a la API de OpenAI directamente desde tu app móvil. Las peticiones pasan por tu Backend (escrito en Python, Node, o Go) donde vive un *Orquestador*. Herramientas como LangChain, LlamaIndex o tus propios scripts manejan el flujo.

2. **Gestión de la Memoria (Persistencia):**
Como el LLM no tiene memoria, tu base de datos debe guardar cada mensaje del usuario. Antes de enviar la pregunta número cinco a la API, el orquestador concatena la pregunta actual con el historial previo y se lo inyecta todo al LLM. 

3. **Capa de Recuperación de Información (RAG / Vector DB):**
Para que el modelo no alucine y sepa datos privados de tu empresa (como tus políticas internas), conectamos una Base de Datos Vectorial (como Pinecone o Chroma). Cuando el usuario pregunta, el sistema busca en la base de datos, extrae el documento correcto y se lo "pasa a leer" al LLM.

4. **Capa de Herramientas y Agentes (Tools / Actions):**
Permitimos que el LLM ejecute código. Le damos acceso a "Herramientas" (APIs). Si le preguntas el clima, el LLM no lo inventa; decide ejecutar una API meteorológica, lee el resultado y te lo explica.

### [02:20-03:20] IDENTIFICACIÓN DE CUELLOS DE BOTELLA
**[VISUAL: Señales de advertencia rojas sobre los diferentes componentes del diagrama]**

Como arquitecto, debes identificar dónde se va a romper tu sistema. Los tres grandes riesgos son:

**1. Cuello de botella de Red (Network Latency):** Llamar a una API externa toma entre 1 y 5 segundos. Si tu app está acostumbrada a tiempos de respuesta de 50 milisegundos, tus usuarios pensarán que la app se colgó. Debes usar flujos asíncronos y respuestas en streaming (como ChatGPT, que escribe letra por letra).

**2. Dependencia de Proveedor Sencillo (Single Point of Failure):** Si Azure OpenAI se cae, tu negocio se detiene. Debes tener un patrón de "Fallback" (respaldo) arquitectónico. Si el LLM primario falla, tu código debe redirigir el tráfico automáticamente a Gemini o Anthropic.

**3. Inyecciones de Prompt (Prompt Injection):** El mayor riesgo de seguridad. Los usuarios maliciosos intentarán engañar a tu LLM para que ejecute comandos prohibidos o revele tu prompt interno (System Prompt). Debes colocar un filtro de seguridad antes de que el texto toque tu modelo.
