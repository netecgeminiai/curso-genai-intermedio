### [00:08-01:00] INTRODUCCIÓN Y EL PROBLEMA DEL JSON
**[VISUAL: Pantalla de código. Un bloque JSON roto (faltan comillas) en color rojo]**

Hola. Hay dos verdades absolutas cuando construyes aplicaciones con IA Generativa en producción.
La primera: las APIs de terceros van a fallar.
La segunda: la IA va a intentar romper tu formato de datos.

Imagina que le pides a ChatGPT que extraiga el nombre y el correo de un CV y te lo devuelva en formato JSON. A veces te devolverá un JSON perfecto. Otras veces dirá: *"¡Claro, aquí tienes tu JSON!: ```json..."*. 

Esa simple frase extra rompe tu código en Python inmediatamente con un `JSONDecodeError`. Hoy vamos a arreglar esto usando **Pydantic** y Salidas Estructuradas.

---

### [01:00-02:15] OBLIGANDO LA ESTRUCTURA CON PYDANTIC
**[VISUAL: Código en Python definiendo una clase BaseModel de Pydantic]**

Pydantic es una librería de Python que nos permite definir cómo deben verse nuestros datos, validarlos y forzar sus tipos.

En lugar de rogarle al modelo en el prompt: *"Por favor, devuelve un JSON con las llaves 'nombre' y 'edad'"*, usamos las capacidades modernas de las APIs (como el `response_format` de OpenAI o el esquema de Gemini) y le pasamos nuestra clase de Pydantic.

```python
from pydantic import BaseModel

class UserProfile(BaseModel):
    name: str
    age: int
    is_active: bool
```

Al pasar este esquema directamente a la llamada de la API, el modelo está *matemáticamente obligado* a generar los tokens en ese orden y formato. Tu backend en Python recibirá un objeto limpio y validado, sin alucinaciones de formato, listo para inyectarse en tu base de datos.

### [02:15-03:20] RATE LIMITS (LÍMITES DE TASA)
**[VISUAL: Icono de un velocímetro en rojo. Un servidor rechazando conexiones: HTTP 429]**

El segundo gran problema en producción son los **Rate Limits**.
Los proveedores de IA limitan cuántas peticiones (Requests per Minute - RPM) y cuántos tokens (Tokens per Minute - TPM) puedes procesar. 

Si tu aplicación se vuelve viral y mil usuarios preguntan al mismo tiempo, la API te devolverá un hermoso error HTTP 429: "Too Many Requests".

Si no manejas este error, tu aplicación se caerá por completo y el usuario verá una pantalla blanca.

### [03:20-04:20] REINTENTOS EXPONENCIALES (EXPONENTIAL BACKOFF)
**[VISUAL: Gráfico de barras escalonadas. El tiempo de espera aumenta (1s, 2s, 4s, 8s)]**

Para solucionar el error 429, implementamos un patrón arquitectónico llamado **Reintento Exponencial** (Exponential Backoff), típicamente usando librerías como `Tenacity` en Python.

La lógica es simple pero poderosa:
1. Tu código hace la llamada a la API.
2. Si falla por límite de tasa, el código "atrapa" el error y se duerme por 1 segundo.
3. Lo intenta de nuevo. Si vuelve a fallar, espera 2 segundos. Luego 4. Luego 8.

```python
from tenacity import retry, wait_exponential, stop_after_attempt

@retry(wait=wait_exponential(multiplier=1, max=10), stop=stop_after_attempt(5))
def call_llm_safely(prompt):

```

Esto le da tiempo a los servidores de OpenAI o Google para "enfriarse" y recargar tu cuota, sin que tu usuario reciba un mensaje de error feo en la pantalla. Simplemente verá un indicador de carga por unos segundos más.
