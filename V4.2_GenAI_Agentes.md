### [00:08-01:00] DE LA RESPUESTA A LA ACCIÓN
**[VISUAL: Pantalla oscura. Un chat de texto: "Busca mi vuelo". Un RAG normal responde: "No lo sé". Un Agente responde: "Buscando... Vuelo cancelado. Te reservé uno nuevo". Título: "Agentes Autónomos"]**

Hola, arquitectos. Hemos llegado a la frontera actual de la Inteligencia Artificial: **Los Agentes Autónomos**.

Imagina un sistema RAG tradicional como un excelente bibliotecario. Le preguntas algo, va a los libros, y te trae la respuesta. Es inteligente, pero es pasivo. No puede salir de la biblioteca.

Un **Agente**, en cambio, es un empleado digital. Le das un objetivo y él decide, por sí mismo, qué pasos tomar y qué herramientas utilizar para cumplirlo. 

Si el usuario escribe *"Cancela mi suscripción y envíame la factura por correo"*, un Agente tiene la autonomía para razonar: *"Paso 1: Entrar a la base de datos SQL y buscar al usuario. Paso 2: Ejecutar la API de Stripe para cancelar. Paso 3: Ejecutar la API de SendGrid para enviar el correo"*.

### [01:00-02:15] LA ARQUITECTURA DE UN AGENTE (EL BUCLE ReAct)
**[VISUAL: Diagrama del bucle ReAct: PENSAMIENTO -> ACCIÓN -> OBSERVACIÓN]**

Para que un LLM (como GPT-4) se convierta en un Agente, lo programamos usando un patrón arquitectónico llamado **ReAct (Razonamiento y Acción)**.

El ciclo funciona así:
1. **Pensamiento (Thought):** El modelo analiza el prompt del usuario y escribe en su memoria interna: *"Para cancelar la suscripción, primero necesito saber el ID del usuario en Stripe"*.
2. **Acción (Action):** El modelo le dice a tu código Python: *"Ejecuta la herramienta de búsqueda en Stripe con el correo juan@ejemplo.com"*.
3. **Observación (Observation):** Tu código ejecuta la API de Stripe, recibe el ID del usuario, y se lo inyecta de vuelta al modelo.

El modelo lee la observación, vuelve a "Pensar", y entra en un ciclo infinito (Loop) de Acción-Observación hasta que determina que el problema del usuario está resuelto. Solo entonces, le escribe la respuesta final al usuario.

### [02:15-03:15] HERRAMIENTAS (FUNCTION CALLING / TOOLS)
**[VISUAL: Código en pantalla definiendo una herramienta de Python decorada con `@tool`]**

Pero, ¿cómo sabe el LLM que existe la "API de Stripe" o que puede buscar en la web? 
Aquí entra el concepto de **Function Calling** o **Tools** (Herramientas).

Como desarrollador, tú programas funciones normales en Python. Por ejemplo, una función que consulta el clima, o una que ejecuta comandos SQL. 

La magia ocurre cuando documentas esa función usando Pydantic (como vimos en el Capítulo 2) y se la envías al LLM en el System Prompt. Le dices: *"Oye, aquí tienes una caja de herramientas. Tienes una llamada `consultar_clima` y otra llamada `consultar_sql`. Úsalas si las necesitas"*.

Los modelos modernos (especialmente los de OpenAI y Gemini) están pre-entrenados para reconocer estas herramientas y devolverte un JSON especial pidiéndote que ejecutes la función en su nombre. El LLM es el cerebro; tu código en Python son las manos.

### [03:15-04:20] EL PELIGRO: GOBIERNO DE AGENTES Y SEGURIDAD
**[VISUAL: Señal de peligro. Un robot a punto de presionar un botón rojo que dice "DROP TABLE Users"]**

Darle manos a la IA es increíblemente peligroso. Si programas una herramienta que hace modificaciones en la base de datos (`UPDATE` o `DELETE`), y el usuario le hace un "Prompt Injection" al agente pidiéndole que borre los registros de todos los clientes, tu agente obediente lo hará.

**Reglas de Arquitectura Segura para Agentes:**
1. **Human-in-the-loop (HITL):** Para acciones destructivas (como hacer reembolsos de dinero, enviar correos masivos o borrar datos), el agente NO debe actuar solo. Debe preparar la acción y pausar el flujo hasta que un humano presione el botón de "Aprobar". LangChain soporta esto nativamente.
2. **Herramientas de Solo Lectura (Read-Only Tools):** En tu versión 1.0, dale al agente solo herramientas que busquen información (GET), no herramientas que la modifiquen (POST/DELETE).
3. **Sandboxing:** Si le das a un agente una herramienta para ejecutar código Python arbitrario (Python REPL), siempre debes correr esa herramienta dentro de un contenedor Docker aislado. Jamás en tu servidor principal.
